## Язык программирования

### Общая характеристика

В проекте реализован JavaScript-подобный язык с C-подобным синтаксисом. 
Программа состоит из функций, каждая функция — из последовательности операторов, 
разделённых точкой с запятой. Поддерживаются объявления переменных, присваивания, 
условные операторы, циклы, вызовы встроенных функций, массивы, строки в формате pstr
и векторные операции для массивов фиксированной длины.

Язык поддерживает:

- объявления переменных `let` (с возможной инициализацией);
- присваивания;
- составной оператор (`block`);
- условный оператор `if` / `else`;
- циклы `while`;
- определения функций `function` (грамматика поддерживает множественные декларации, но текущая реализация кодогенератора поддерживает выполнение только функции `main()`; вызовы пользовательских функций не реализованы);
- оператор `halt;` для остановки модели;
- встроенные функции:
  - `print(int)` — выводит младший байт аргумента как ASCII-символ на `OUT_PORT`;
  - `print_num(int)` — выводит **положительное** целое число в десятичном виде (отрицательные числа не поддерживаются);
  - `print_str(int[])` — выводит строку в формате pstr;
  - `read()`, `readln()` — чтение из `IN_PORT`;
  - `len(arr)` — длина массива (вычисляется в момент компиляции, заменяется на литерал);
  - `vload(arr, idx?)` — загружает 4 элемента `arr[idx..idx+3]` в вектор;
  - `vadd(a, b)` — поэлементное сложение двух векторов;
  - `vstore(arr, idx?, vec)` — сохраняет вектор `vec` в `arr[idx..idx+3]`;
- литералы: целые числа, шестнадцатеричные числа, символы, строки, `true`, `false`;
- арифметические, побитовые, логические и операторы сравнения с обычным приоритетом C;
- короткое замыкание `&&` и `||` на этапе трансляции.

Транслятор (`src/hl_logic.py`) реализует подмножество грамматики, достаточное для написания учебных программ, и компилирует исходник в текстовое ассемблерное представление, которое затем ассемблируется в бинарный образ.

### Синтаксис (BNF)

```bnf
<comment>      ::= "//" <any-char>* <newline>
<identifier>   ::= [a-zA-Z_][a-zA-Z0-9_]*
<number>       ::= [0-9]+
<hex-number>   ::= "0x" [0-9a-fA-F]+
<char>         ::= "'" <print-char> "'"
<string>       ::= '"' <print-char>* '"'
<separator>    ::= "{" | "}" | "(" | ")" | "[" | "]" | ";" | ","

<program>      ::= <function>*

<var-decl>     ::= "let" <identifier> ("[" <expr> "]")? ("=" <expr>)?

<function>     ::= "function" <identifier> "(" ")" <block>

<block>        ::= "{" <statement>* "}"

<statement>    ::= <var-decl> ";"
                 | <assignment> ";"
                 | <if-stmt>
                 | <while-stmt>
                 | <return-stmt> ";"
                 | <call-stmt> ";"
                 | <block>
                 | "halt" ";"
                 | ";"

<assignment>   ::= <identifier> ("[" <expr> "]")* "=" <expr>

<if-stmt>      ::= "if" "(" <expr> ")" <statement> ("else" <statement>)?

<while-stmt>   ::= "while" "(" <expr> ")" <statement>

<return-stmt>  ::= "return" <expr>?

<call-stmt>    ::= <identifier> "(" <arg-list>? ")"

<expr>         ::= <literal>
                 | <identifier>
                 | <unary-op> <expr>
                 | <expr> <binary-op> <expr>
                 | <identifier> "(" <arg-list>? ")"
                 | <identifier> ("[" <expr> "]")+
                 | "{" <arg-list>? "}"
                 | "(" <expr> ")"

<literal>      ::= <number> | <hex-number> | <char> | <string> | "true" | "false"

<arg-list>     ::= <expr> ("," <expr>)*

<unary-op>     ::= "-" | "~" | "!"

<binary-op>    ::= "+" | "-" | "*" | "/" | "%"
                 | "<<" | ">>"
                 | "&" | "|" | "^"
                 | "==" | "!=" | "<" | ">" | "<=" | ">="
                 | "&&" | "||"
```

### Семантика

#### Стратегия вычислений

- используется **eager evaluation**: аргументы выражений вычисляются до применения операции;
- `if` вычисляет только выбранную ветвь;
- `&&` и `||` имеют short-circuit семантику: правый операнд не вычисляется, если результат уже определён;
- `while` повторно вычисляет условие и выполняет тело, пока условие истинно;
- `function` вводит новую область видимости;
- `halt;` останавливает модель процессора;
- `len(arr)` — возвращает количество элементов массива (известно на этапе компиляции).

#### Области видимости

- программная единица — функция; глобальной видимости нет (статические объекты в `.data` — это не переменные языка, а размещённые компилятором данные: строковые литералы, инициализированные массивы);
- переменные, объявленные через `let`, видны только внутри блока, в котором они объявлены, и во вложенных блоках;
- имя переменной должно быть объявлено до первого использования;
- параметры функций в текущей реализации не поддерживаются (используется `function main()`).

#### Типизация

Язык использует статическую типизацию с двумя представлениями данных: 
целые числа `int` и статические массивы `int[]`. Строки не являются отдельным типом и хранятся 
как массивы `int[]` в формате `pstr`. Символы и логические значения — это `int` (их значения суть ASCII-код и 0/1).

#### Виды литералов

| Литерал          | Тип значения | Пример          |
|------------------|--------------|------------------|
| численный        | `int`        | знаковое 32-бит целое |
| символьный       | `int`        | `'A'`, `'\n'` (ASCII-код символа) |
| строковый        | `int[]`      | `"Hello"` — массив ASCII-кодов в pstr-формате |
| логические       | `int`        | `true` (1), `false` (0) |
| массив           | `int[]`      | `{1, 2, 3}` — элементы через запятую |

Тип `int` — 32-битное знаковое целое. Символы и логические значения — это тоже `int` (их значения суть ASCII-код и 0/1). Строки и массивы — `int[]`; отдельного типа `char` или `string` нет. Элементами массива могут быть любые `int`-выражения, в том числе символьные литералы: `{'A', 66, '\n'}`.

#### Массивы

Язык поддерживает статические одномерные массивы `int[]`. Массивы размещаются в `gp`-относительной памяти данных. Доступ к элементам осуществляется через индексацию в квадратных скобках.

**Объявление массивов:**

```javascript
// Массив с инициализацией из литерала (размер определяется количеством элементов)
let arr = {10, 20, 30, 40, 50};

// Массив с указанием размера (все элементы обнуляются)
let buf[256];

// Массив с указанием размера и частичной инициализацией
let data[10] = {1, 2, 3};

// Смешанный массив (все элементы — 32-битные слова)
let mixed = {65, 'B', 0x21}; // {'A', 'B', '!'}
```

**Доступ к элементам:**

```javascript
let x = arr[0];       // чтение элемента
arr[1] = 99;          // запись элемента
let i = 2;
let y = arr[i];       // индекс может быть переменной
arr[i + 1] = 42;      // индекс может быть выражением
```

**Статические операции над массивами:**

При применении бинарных операторов (`+`, `-`, `*`, `/`) к двум массивам одинакового размера компилятор генерирует векторные инструкции (`VLD`/`VADD`/`VST` и т.д.), обрабатывающие 4 элемента за раз (SIMD). Результат сохраняется в новый массив.

```javascript
let a = {1, 2, 3, 4};
let b = {5, 6, 7, 8};
let c = a + b;  // генерирует VLD / VADD / VST — c = {6, 8, 10, 12}
```

#### Пример программы


## Организация памяти

### Общая модель

Используется **гарвардская архитектура** (вариант `harv`): память инструкций и память данных физически разделены. Машинное слово — 32 бита. Адресация — пословная, выровненная.

Параметры памяти:

- машинное слово: 32 бита;
- инструкция: 32 бита фиксированной длины;
- адресация словная, alignment 4 байта;
- размер памяти данных: 8192 слова (см. `DATA_MEM_SIZE` в `src/isa.py:32`);
- база стека: `0x2000` (`STACK_BASE = DATA_MEM_SIZE`), стек растёт вниз;
- статические данные располагаются от `0x0001` вверх (`gp = data_start = 1`), стек — от `0x2000` вниз; при коллизии перезапись данных не отслеживается;
- память инструкций (instruction memory): word-addressable, адрес = PC, начинается с `0` (адрес 0 занят `J main`);
- ввод-вывод реализован через memory-mapped I/O (вариант `mem`);
- исключения (деление на ноль, неверный адрес, переполнение стека, illegal instruction) **не обрабатываются** — модель зацикливается или выдаёт некорректный результат. Это учебное упрощение.

### Регистры

#### Скалярные регистры (32 × 32 бита)

| Алиас   | Номер | Назначение                  |
|---------|------:|------------------------------|
| `zero`  | 0     | всегда 0                    |
| `ra`    | 1     | адрес возврата              |
| `sp`    | 2     | указатель стека             |
| `gp`    | 3     | база статических данных (frame)  |
| `a0..a7`| 4..11 | зарезервированы для аргументов и возврата (в текущей версии HL пользовательские функции без параметров; регистры доступны как временные)|
| `t0..t5`| 12..17| временные                   |
| `s0..s11`| 18..29| автоматическое распределение (имена унаследованы от RISC-V, но ABI не реализован — ни один регистр не сохраняется вызываемым кодом) |
| `t6`    | 30    | временный                   |
| `tp`    | 31    | резерв                      |

#### Vector-регистры (8 × 128 бит, вариант `vector`)

| Алиас | Размер         | Назначение                                          |
|-------|----------------|------------------------------------------------------|
| `V0..V7` | 4 lane × int32 | vector register file для VLD/VST и lane-wise ALU  |

Служебный регистр datapath для vector-расширения:

| Регистр         | Назначение                                    |
|-----------------|-----------------------------------------------|
| `vbase`         | базовый адрес для последовательного VLD/VST    |

### Раздельные адресные пространства (Harvard)

Память инструкций и память данных **не пересекаются**. Адресация каждой — пословная, начиная с 0. MMIO-регистры `IN_PORT`/`OUT_PORT` отображаются только в пространство данных.

#### Instruction memory (отдельное адресное пространство)

```text
адрес   +------------------------+
0       | J main                 |
1       | main: ...              |
2       | функции пользователя   |
3       | сгенерированные        |
4       | вспомогательные коды   |
...     +------------------------+
```

PC адресует инструкционную память. Адрес = номер слова (инкремент `PC` на `+1` за такт). Размер = число эмитированных инструкций.

#### Data memory (отдельное адресное пространство)

```text
0x0001 - ...      статические данные (gp-относительно, .word, .string)
...    - 0x1FFF   стек (растёт вниз от 0x2000)
0xFFFFFFF0        IN_PORT (чтение)
0xFFFFFFF4        OUT_PORT (запись)

Пример содержимого (gp = 1):
адрес   +------------------------------+
1       | .word 5 (length)             | <--- "Hello" pstr (gp+0)
2       | .word 72 ('H')               |
3       | .word 101 ('e')              |
...     | ...                          |
7       | .word 0 (dummy/padding)      |
8       | .word 10 (arr[0])            | <--- let arr = {10, 20...}
9       | .word 20 (arr[1])            |
...     +------------------------------+
0x1FFF  | стек                         |
        +------------------------------+
0xFFF0  +------------------------------+
        | IN_PORT — чтение             |
0xFFF4  +------------------------------+
        | OUT_PORT — запись            |
        +------------------------------+
```
 
#### Memory-mapped I/O (вариант `mem`)
 
| Адрес        | Имя       | Доступ     | Назначение                              |
|--------------|-----------|------------|------------------------------------------|
| `0xFFFFFFF0` | `IN_PORT` | read-only  | MMIO: читает следующий байт из входного потока |
| `0xFFFFFFF4` | `OUT_PORT`| write-only | MMIO: пишет младший байт в выходной поток      |
 
Эти адреса находятся вне диапазона физической памяти данных (за пределами `0x1FFF`) и перехватываются моделью `DataMemory`. Запись в `IN_PORT` или чтение из `OUT_PORT` в модели не приводит к ошибке, но и не имеет наблюдаемого эффекта. В ассемблере адреса `IN_PORT`/`OUT_PORT` указываются как `0xFFF0`/`0xFFF4`. Immediate-поле инструкций `LW`/`SW` имеет размер 11 бит со знаком, поэтому полное значение `0xFFFFFFF0` не может быть закодировано напрямую: ассемблер берёт младшие 11 бит (`0x7F0` = 2032 = −16 в sign-11), которые процессор затем знаково-расширяет до 32 бит, восстанавливая `0xFFFFFFF0`. Аналогично `0xFFF4` → `0x7F4` (−12) → `0xFFFFFFF4`.

### Формат строк `pstr`

Строка в runtime хранится как последовательность 32-битных слов с префиксом длины:

```text
addr + 0  : length
addr + 1  : char[0]
addr + 2  : char[1]
...
```

Каждый символ занимает одно машинное слово; `length` хранит количество символов. Директива `.string "Hello"` в ассемблере автоматически разворачивается в 6 слов.

### Формат массивов `int[]` в HL

Каждый массив занимает два последовательных участка в статической области данных (`gp`-относительно):

1. **Слот-указатель** (1 слово) — хранит абсолютный адрес первого элемента массива.
2. **Данные массива** (`sz` слов) — элементы массива.

Массивы с литеральным инициализатором (`let arr = {1, 2, 3}`) размещаются **статически** в секции `.data`: значения элементов записываются как `.word` в бинарном образе и загружаются в память данных вместе с программой, без heap-аллокации и без runtime-инициализации:

```text
.data:
  gp + slot      : <ptr>          ; указатель на данные (значение = адрес gp+data_off)
  gp + data_off  : elem[0]        ; первый элемент
  gp + data_off+1: elem[1]
  gp + data_off+2: elem[2]
  ...
```

Массивы без инициализатора (`let buf[256]`) также размещаются статически: все элементы обнуляются (`.word 0`).

Доступ к элементу: `LW ptr, gp, slot` → загрузка адреса данных; `ADD ptr, ptr, idx` → вычисление адреса элемента; `LW val, ptr, 0` → чтение.

Длина известна на этапе компиляции и хранится во внутренних структурах транслятора. Вызов `len(arr)` возвращает константу времени компиляции и не порождает инструкций. Проверка границ массива в runtime не производится.

При векторных операциях (`a + b`, `a - b`, `a * b`, `a / b`) транслятор проверяет равенство длин операндов; векторизация (strip-mining) применяется для массивов длины ≥ 4: каждые 4 элемента обрабатываются VLD/VADD/VST, остаток — скалярно.

### Механика отображения объектов языка на память

**Литералы.** `42` → `ADDI rd, zero, 42`. Если вне диапазона `-1024..1023`: `0x12345` → `LUI rd, 0x12345 >> 11` + `ADDI rd, rd, 0x12345 & 0x7FF`. Символьный литерал `'A'` → `ADDI rd, zero, 65`. `true`/`false` → `1`/`0`. Строковый литерал `"Hello"` → `.string` в `.data` (длина + символы).

**Переменные.** Первые 12 имён → `s0..s11`. Остальные → статическая область данных (доступ через `gp + offset`): `let x = 5` → `ADDI t0, zero, 5` + `SW t0, gp, offset`.

**Массивы.** `let arr = {10, 20, 30}` → в `.data` выделяются слот-указатель (хранит адрес `gp+data_off`) и 3 слова `.word 10, 20, 30`. Доступ `arr[i]` → `LW ptr, gp, slot` (загрузка адреса данных), `ADD ptr, ptr, idx`, `LW rd, ptr, 0` (чтение элемента).

**Инструкции.** Кодируются ассемблером в 32-битные слова. Код начинается с адреса 0 инструкцией `J main`; остальные инструкции располагаются последовательно в instruction memory согласно результатам ассемблирования. Метки разрешаются за два прохода.

**Процедуры.** `function main()` — единственная точка входа; runtime-процедуры (`print_str`, `print_num`) компилируются inline.

### Формат вывода

Вывод программы формируется через `SW` в `OUT_PORT` (адрес `0xFFFFFFF4`). Каждая запись выводит **младший байт** (0–255) как ASCII-символ:

| Функция | Формат вывода | Пример |
|---------|--------------|--------|
| `print(x)` | Младший байт `x` как ASCII | `print(65)` → `'A'` |
| `print_num(x)` | Десятичное представление `x` (цифра за цифрой) | `print_num(42)` → `'42'` |
| `print_str(s)` | Строка в pstr-формате | `print_str("Hi")` → `'Hi'` |

Символы с кодами 0–255 выводятся как есть. Перевод строки — `print(10)` (ASCII `\n`). Пробел — `print(32)`. `print_num` автоматически преобразует целое число в последовательность цифр через многократное деление на 10.

## Система команд

### Особенности процессора

- **RISC** ISA (load/store): арифметика работает только над регистрами, память доступна только через `LW`/`LB`/`SW`/`SB`/`LUI`. Название RISC относится к свойствам системы команд (фиксированная длина, небольшое число форматов), при этом реализация управляющего устройства является микропрограммной.
- **Harvard** (вариант `harv`): инструкции и данные в разных адресных пространствах.
- **Фиксированная длина** инструкции: 32 бита.
- **Микрокодированная** CU (вариант `mc`): горизонтальная µROM с 25 сигналами.
- **Tick-accurate** (вариант `tick`): каждый вызов `tick()` моделирует ровно одну микроинструкцию.
- **Двоичное** представление (вариант `binary`): выход ассемблера — поток 32-битных little-endian слов.
- **Stream I/O** (вариант `mem`): `IN_PORT`/`OUT_PORT` в memory map.
- **Pascal strings** (`pstr`): `pstr`-формат хранения строк в data memory.
- **SIMD extension** (вариант `vector`): 8 регистров по 4 lane × int32. **Примечание**: память данных является **однопортовой** для слов (single-word MDR), поэтому векторные загрузка/сохранение выполняются последовательно.

### Форматы инструкций

Все инструкции имеют длину 32 бита.

| Формат | Назначение                                                  |
|--------|--------------------------------------------------------------|
| R      | регистр-регистр арифметика                                  |
| I      | immediate, `LW`/`LB`                                        |
| S      | `SW`/`SB` (I-подобный: `rd` не используется, `rs2` в поле `0`) |
| B      | условные переходы                                            |
| U      | `LUI` (загрузка верхней части immediate)                    |
| J      | `J` (безусловный переход)                                   |
| JL     | `JAL` (переход с сохранением PC+1 в регистр)               |
| JR     | `JR` (косвенный переход через регистр)                      |
| V      | Vector ALU (`VADD`/`VSUB`/`VMUL`/`VDIV`/`VCMP`)             |
| VL     | Vector Load/Store (`VLD`/`VST`)                             |
| —      | `HALT` (опкод без операндов)                                |

#### R-type

```text
    31      26 25    21 20    16 15    11 10         0
   +--------+--------+--------+--------+-----------+
   | opcode |   rd   |  rs1   |  rs2   |     0     |
   +--------+--------+--------+--------+-----------+
   |  6 бит |  5 бит |  5 бит |  5 бит |   11 бит  |
```

Используют: `ADD`, `SUB`, `MUL`, `DIV`, `REM`, `MULH`, `AND`, `OR`, `XOR`, `NOT`, `SLL`, `SRL`, `SRA`, `SLT`, `NOP`.

#### I-type

```text
    31      26 25    21 20    16 15    11 10         0
   +--------+--------+--------+--------+-----------+
   | opcode |   rd   |  rs1   |   0    |    imm    |
   +--------+--------+--------+--------+-----------+
   |  6 бит |  5 бит |  5 бит |  5 бит |   11 бит  |
```

Используют: `ADDI`, `ANDI`, `ORI`, `XORI`, `SLLI`, `SRLI`, `SRAI`, `SLTI`, `LW`, `LB`.

> **Важно:** `SW`/`SB` имеют тот же размер полей, что и I-type, но с другой интерпретацией: `rd` (биты 21–25) игнорируется (зарезервирован), а поле `0` (биты 11–15) содержит регистр-источник `rs2`. Формат S-type: `opcode(6) | 0(5) | rs1(5) | rs2(5) | imm(11)`. B-type использует `rs1` и `rs2` для операндов сравнения, а `imm` — знаковое PC-относительное смещение.

#### B-type

```text
    31      26 25    21 20    16 15    11 10         0
   +--------+--------+--------+--------+-----------+
   | opcode |   0    |  rs1   |  rs2   |  offset   |
   +--------+--------+--------+--------+-----------+
   |  6 бит |  5 бит |  5 бит |  5 бит |   11 бит  |
```

Используют: `BEQ`, `BNE`, `BLT`, `BLE`, `BGT`, `BGE`, `BGTU`, `BLEU`, `BLTU`, `BGEU`.

Offset — знаковое, 11-битное, в словах: диапазон `-1024..+1023` от PC.

#### U-type

```text
    31      26 25    21 20                             0
   +--------+--------+-------------------------------+
   | opcode |   rd   |       imm (21 бит)            |
   +--------+--------+-------------------------------+
   |  6 бит |  5 бит |           21 бит              |
```

Использует: `LUI` (`rd = imm[20:0] << 11`). Immediate 21 бит, сдвиг на 11 (а не на 12, как в RISC-V), чтобы пара `LUI` + `ADDI` покрывала весь 32-битный диапазон: `(imm21 << 11) + sext(imm11)`. Ассемблер автоматически корректирует старшую часть: если младшие 11 бит константы устанавливают знаковый бит (`>= 0x400`), то `imm21` увеличивается на 1, чтобы компенсировать отрицательное расширение знака `ADDI`.

#### J / JL / JR

```text
   J:  [ opcode (6) |          target (26)             ]
   JL: [ opcode (6) | rd (5) |     target (21)         ]
   JR: [ opcode (6) | rm (5) |           0             ]
```

#### V-type / VL-type

```text
    31      26 25    21 20    16 15    11 10         0
   +--------+--------+--------+--------+-----------+
   | opcode |   Vd   |  Vs1   |  Vs2   |  reserved |
   +--------+--------+--------+--------+-----------+
   |  6 бит |  5 бит |  5 бит |  5 бит |   11 бит  |
```

Для `VLD`/`VST`: `Vs1` = базовый GPR, младшие 11 бит = signed offset.

### Детальная реализация по типам (Маппинг микрокоманд)

Микропрограммы (µROM, `src/control_path.py`) используют следующие последовательности:

* **R-type:** `FETCH → R_EX → FETCH`
* **I-type (arith):** `FETCH → I_EX → FETCH`
* **Load (`LW`/`LB`):** `FETCH → L_EX1 → L_EX2 → L_EX3 → FETCH`
* **Store (`SW`/`SB`):** `FETCH → S_EX1 → S_EX2 → FETCH`
* **Branch (B-type):** `FETCH → B_EX → FETCH`
* **Jump (`J`):** `FETCH → J_EX → FETCH`
* **Jump and Link (`JAL`):** `FETCH → JL_EX1 → JL_EX2 → FETCH`
* **Indirect Jump (`JR`):** `FETCH → JR_EX → FETCH`
* **LUI:** `FETCH → U_EX → FETCH`
* **Vector ALU:** `FETCH → V_EX → FETCH`
* **Vector Load (`VLD`):** `FETCH → VLD_EX → VLD_W0 → VLD_W1 → VLD_W2 → VLD_W3 → FETCH`
* **Vector Store (`VST`):** `FETCH → VST_EX1 → VST_W0 → VST_W1 → VST_W2 → VST_W3 → FETCH`
* **NOP:** `FETCH → NOP_EX → FETCH`
* **HALT:** `FETCH → HALT_EX`

### Точность моделирования

`Machine.tick()` выполняет ровно одну микроинструкцию: читает текущую `MI` через `Microsequencer.current_mi(IR)`, передаёт её в `DataPath.tick(...)`, делает snapshot, затем вызывает `Microsequencer.advance(...)` для обновления `uPC`.

Журнал (`get_journal`) для каждого такта содержит:

- номер такта;
- `uPC`, `MIR` (компактное 64-битное представление микро-инструкции);
- `PC`, `IR`;
- `Micro` (имя микроинструкции);
- активные сигналы (`ir_we`, `pc+1`, `mar_we`, `mem_rd`, `mem_wr`, `reg_we`);
- мнемонику декодированной инструкции;
- значения 16 видимых регистров (`ZERO`..`T3` в текущем snapshot'е, плюс `AR`/`DR`);
- накопленный выход модели.

### Программная модель

Расширение добавляет 8 векторных регистров `V0..V7`:
- Размер регистра = **128 бит**
- Количество lane = **4**
- Размер lane = **32 бита**

За одну операцию обрабатываются сразу 4 элемента:

- `VLD`/`VST` — загрузка/запись 4 последовательных 32-битных слов из памяти;
- `VADD`/`VSUB`/`VMUL`/`VDIV`/`VCMP` — lane-wise ALU за один такт.

> Векторное ALU обрабатывает 4 lane за 1 такт, однако полного ускорения 4× не достигается из-за того, что VLD и VST выполняют последовательный доступ к однословной памяти данных (по одному слову за такт). Арифметика ускоряется в 4×, но загрузка/сохранение остаётся узким местом.

Длина векторного регистра фиксирована (4 lane). HL-компилятор поддерживает массивы **любой длины** $N$ (не только 4). Операция вида `let c = a + b` для массивов длиной $N$ автоматически векторизуется через **strip-mining**:

1.  **Векторные блоки**: Массив разбивается на $N // 4$ блоков по 4 элемента. Для их обработки генерируется цикл с инструкциями `VLD`, `VADD` (или другими векторными опкодами) и `VST`. Внутри цикла указатели адресов инкрементируются на 4 за итерацию.
2.  **Скалярный хвост (Scalar Tail)**: Оставшиеся $N \% 4$ элементов (хвост от 1 до 3 штук, включая нечётные размеры) обрабатываются последовательно обычными скалярными инструкциями `LW`, `ADD`, `SW`.
3.  **Fast Path**: Если $N=4$, цикл не создается — генерируется прямой поток векторных команд. Если $N < 4$, используется только скалярный код.

Это позволяет прозрачно использовать векторное ускорение для массивов произвольного размера без ручного управления циклами.

Встроенные функции `vload(arr, idx)` и `vstore(arr, idx, vec)` принимают опциональный индекс для загрузки/сохранения 4 элементов с произвольной позиции. По умолчанию `idx = 0`.

### Аппаратные блоки

| Блок                       | Назначение                                    |
|----------------------------|-----------------------------------------------|
| `VectorRegisterFile` (VRF) | 8 регистров × 4 lane × int32                 |
| `VectorALU`                | lane-wise ALU                                 |
| `vbase`                    | базовый адрес для последовательного VLD/VST   |

### Демонстрация: scalar vs vector

В варианте `vector` реализована SIMD-обработка массивов из 4 элементов. Сравнение двух реализаций одной и той же задачи (сложение двух 4-элементных массивов, печать результата `11 22 33 44`):

- **Scalar HL** (`examples/testNoVector.alg`): классический цикл с условными переходами, скомпилирован в `LW`/`ADD`/`SW`.
- **HL Vector** (`examples/testVector.alg`): синтаксис массивов (`let a = {1,2,3,4}`, `let c = a + b`), компилятор генерирует `VLD`/`VADD`/`VST`.

`scalar loop` (см. `examples/testNoVector.alg`):

```alg
function main() {
    let i = 0;
    let a = 0;
    let b = 0;
    let c = 0;
    let tens = 0;
    let ones = 0;

    while (i < 4) {
        if (i == 0) { a = 1;  b = 10; }
        if (i == 1) { a = 2;  b = 20; }
        if (i == 2) { a = 3;  b = 30; }
        if (i == 3) { a = 4;  b = 40; }

        c = a + b;
        tens = c / 10;
        ones = c % 10;

        print_num(tens);
        print_num(ones);
        print(32);

        i = i + 1;
    }

    print(10);
    halt;
}

```

`vector` (см. `examples/testVector.alg`):

```alg
function main() {
    let a = {1, 2, 3, 4};
    let b = {10, 20, 30, 40};
    let c = a + b;
    let i = 0;
    while (i < 4) {
        let tens = c[i] / 10;
        let ones = c[i] % 10;
        print_num(tens);
        print_num(ones);
        print(32);
        i = i + 1;
    }
    print(10);
    halt;
}

```

Golden-тест `vector_demo` запускает hand-written asm-версию и проверяет, что вывод равен `11 22 33 44\n`.

#### Сравнение производительности (tick-accurate замер)

Измерение выполнено на задаче сложения двух массивов из 4 элементов и печати результата (`11 22 33 44`). Один **такт** соответствует одной микростадии µROM (один вызов `Machine.tick()`).

| Метрика                              | Scalar (HL) | HL Vector |
| ------------------------------------ | ----------: | --------: |
| Микро-тактов до `HALT`               | 414         | 386       |
| Ускорение относительно scalar        | 1×          | **~1.07×**|
| Выход программы                      | `11 22 33 44 \n` | `11 22 33 44 \n` |

> Ускорение ~7%, так как бóльшая часть тактов приходится на `print_num` (деление на 10 в цикле), а не на сложение массивов. Ядро — сложение двух 4-элементных массивов — в скалярном варианте: загрузка 4 элементов левого массива (4 × 4 такта `LW`) + загрузка 4 правого (4 × 4) + 4 сложения (4 × 2) + 4 сохранения (4 × 3) + управление циклом ≈ 64 такта. В SIMD: `VLD` (6) + `VLD` (6) + `VADD` (2) + `VST` (6) = 20 тактов, то есть ускорение самого вычисления ~3.2×. Общий выигрыш мал из-за доминирования `print_num`. Для массивов длиной >4 компилятор генерирует strip-mined loop: по 4 элемента за итерацию с помощью VLD/VADD/VST + scalar tail для остатка.


 
### Выполнение `VLD`

Всего инструкция `VLD` занимает 6 тактов: `FETCH`, `VLD_EX` и четыре последовательных стадии `VLD_W0..VLD_W3`:

1. `FETCH + VLD_EX`: `addr = rs1 + imm` → `MAR`, `vbase = MAR`.
2. `VLD_W0`: `MDR = MEM[vbase]` → `VRF[rd][0]`, `vbase++`.
3. `VLD_W1`: `MDR = MEM[vbase]` → `VRF[rd][1]`, `vbase++`.
4. `VLD_W2`: `MDR = MEM[vbase]` → `VRF[rd][2]`, `vbase++`.
5. `VLD_W3`: `MDR = MEM[vbase]` → `VRF[rd][3]`, переход к `Fetch`.

Несмотря на SIMD-вычисление, память данных остаётся однопортовой (single-port, single-word MDR), поэтому загрузка четырёх lane выполняется последовательно — по одному слову за такт.

### Выполнение `VST`

1. `Fetch + VST_EX1`: `addr = rs1 + imm` → `MAR`, `vbase = MAR`.
2. `VST_W0` → `VST_W1` → `VST_W2` → `VST_W3`: каждый такт пишет один lane (`VRF[rd][i]`) в `MEM[vbase]` и инкрементирует `vbase`. 4 lane записываются последовательно за 4 такта.





## Транслятор

Транслятор реализован как консольное приложение, обеспечивающее полный цикл преобразования исходного кода в исполняемый бинарный образ.

### Принципы работы

Транслятор работает в два этапа:

1. **Компиляция (HL Compiler - `src/hl_logic.py`)**:
   - **Lexer**: Разбиение исходного кода (`.alg`) на токены.
   - **Parser**: Построение абстрактного синтаксического дерева (AST).
   - **CodeGen**: Обход AST и генерация ассемблерного кода (`.asm`) с использованием пред-аллоцированных регистров.

2. **Ассемблирование (Assembler - `src/translator.py`)**:
   - Принимает ассемблерный код (`.asm`) и преобразует его в машинный код (`.bin`).
   - Разрешает метки переходов за два прохода (First pass: вычисление адресов, Second pass: кодирование инструкций).
   - Генерирует файл `.lst` для отладки.

### Интерфейс командной строки

Для взаимодействия используется `src/cli.py`.

#### Команда `compile` (полная трансляция)

```bash
python -m src.cli compile <input.alg> <output.bin>
```

- **Входные данные:**
  - `<input.alg>`: имя файла с исходным кодом.
- **Выходные данные:**
  - `<output.bin>`: итоговый бинарный файл.

#### Команда `asm` (только ассемблирование)

```bash
python -m src.cli asm <input.asm> <output.bin>
```

- **Входные данные:**
  - `<input.asm>`: имя файла с ассемблерным кодом.
- **Выходные данные:**
  - `<output.bin>`: итоговый бинарный файл.

## Модель процессора

### Общая структура

Архитектура **не является конвейерной** — процессор выполняет микропрограмму строго последовательно. В каждый такт выполняется ровно одна микроинструкция. Управление потоком микрокоманд и переходы между ними осуществляются посредством микросеквенсора, управляющего регистром `uPC` (Micro Program Counter).

Модель состоит из:

- **DataPath** (`src/data_path.py`): register file, vector register file, ALU, branch comparator, data memory, program counter, instruction register, MAR/MDR, ALU_OUT, feedback bus, vbase;
- **ControlPath** (`src/control_path.py`): горизонтальная µROM на 25 сигналов, микросеквенсор, ALU-декодер, dispatch-логика;
- **DataMemory** (`src/data_path.py:116`): слово-адресуемая память данных 8 KiB, с перехватами `IN_PORT`/`OUT_PORT`;
- **InstructionMemory**: массив слов, адресуется `PC`;
- **Машина** (`src/machine.py`): объединяет DP+CP, ведёт журнал тактов, ограничивает `max_ticks`.

### DataPath

`DataPath` содержит:

- `PC`, `IR`;
- `RegisterFile` (32 × 32 бит, читается по `rs1`/`rs2`, пишется по `rd` через `reg_we` + `reg_src`);
- `VectorRegisterFile` (8 × 4 × 32 бит, поддерживает запись целого регистра и одного lane);
- `ALU`: `ADD`/`SUB`/`MUL`/`MULH`/`DIV`/`REM`/`AND`/`OR`/`XOR`/`NOT`/`SLL`/`SRL`/`SRA`/`SLT`;
- `BranchComparator`: возвращает `eq`, `lt`, `ltu` для `rs1`/`rs2`;
- `MAR`, `MDR`, `ALU_OUT`, `feedback_bus`;
- `vbase` (vector base register);
- `DataMemory`: слова, отображение `IN_PORT`/`OUT_PORT` на input/output stream.

Основные мультиплексоры:

- `MUX_A`: `0=none`, `1=rs1`, `2=PC` (для `B_EX`: `PC + imm`);
- `MUX_B`: `0=none`, `1=rs2`, `2=imm_s`, `3=zero`;
- `MUX_WB` (`reg_src`): `0=none`, `1=ALU_OUT`, `2=MDR`, `3=PC+1`, `4=imm<<11`, `5=imm_u26`, `6=imm_u21`;
- `MUX_PC`: `PC+1` либо `feedback_bus` (`pc_src`);
- `MUX_VBASE`: `MAR` либо `vbase+1` (`vbase_sel`);
- `MUX_ADDR`: `MAR` либо `vbase` (`addr_sel`);
- `MUX_MEM_DATA`: `rs2` либо выход `VRF` (`mem_data_src`).

![DataPath](scheme/scheme_DATAPATH.jpg)

### ControlUnit

`ControlUnit` — горизонтально-микрокодированный. µROM содержит 25 управляющих сигналов. На каждом такте выбирается микроинструкция по текущему адресу `uPC`; после её исполнения значение `uPC` обновляется в соответствии с полем `br_type`:

| `br_type` | Следующий `uPC`                                  |
|-----------|---------------------------------------------------|
| `0`       | `uPC + 1` (следующий шаг)                         |
| `1`       | `uPC = mi.addr` (явный переход)                   |
| `2`       | dispatch по формату инструкции (R/I/L/S/B/J/...)  |
| `3`       | `uPC = 0` (возврат в `FETCH`)                    |

Микроинструкция `FETCH` (адрес 0) инициирует цикл выборки: устанавливаются сигналы `ir_we = 1` и `pc_inc = 1`, после чего `br_type = 2` выполняет диспетчеризацию к начальному адресу микропрограммы конкретной инструкции на основе её формата.

Перечень микроинструкций в µROM (28 элементов, индексы 0–27):

```
FETCH (0),     R_EX (1),      I_EX (2),      L_EX1 (3),
L_EX2 (4),     S_EX1 (5),     B_EX (6),      J_EX (7),
JL_EX1 (8),    JR_EX (9),     U_EX (10),     NOP_EX (11),
HALT_EX (12),  V_EX (13),     VLD_EX (14),   (reserved) (15),
VST_EX1 (16),  L_EX3 (17),    S_EX2 (18),    VLD_W3 (19),
VST_W0 (20),   VST_W1 (21),   VST_W2 (22),   VST_W3 (23),
JL_EX2 (24),   VLD_W0 (25),   VLD_W1 (26),   VLD_W2 (27)
```

Каждая микроинструкция — один такт модели. Даже `FETCH` занимает 1 такт.

### Микрокодирование

| Инструкция          | µROM-последовательность                                                          |
|---------------------|----------------------------------------------------------------------------------|
| `ADD`/`SUB`/...     | `FETCH → R_EX → FETCH`                                                           |
| `ADDI`/`LUI`/...    | `FETCH → I_EX → FETCH` (или `FETCH → U_EX → FETCH` для `LUI`)                  |
| `LW`/`LB`           | `FETCH → L_EX1 → L_EX2 → L_EX3 → FETCH`                                          |
| `SW`/`SB`           | `FETCH → S_EX1 → S_EX2 → FETCH`                                                  |
| `BEQ`/`BNE`/...     | `FETCH → B_EX → FETCH`                                                           |
| `J`                 | `FETCH → J_EX → FETCH`                                                           |
| `JAL`               | `FETCH → JL_EX1 → JL_EX2 → FETCH`                                                |
| `JR`                | `FETCH → JR_EX → FETCH`                                                          |
| `VADD`/`VSUB`/...   | `FETCH → V_EX → FETCH` (vector ALU сразу считает все 4 lane)                   |
| `VLD`               | `FETCH → VLD_EX → VLD_W0 → VLD_W1 → VLD_W2 → VLD_W3 → FETCH` |
| `VST`               | `FETCH → VST_EX1 → VST_W0 → VST_W1 → VST_W2 → VST_W3 → FETCH`                    |
| `HALT`              | `FETCH → HALT_EX`                                                                |

### Управляющие сигналы (microinstruction fields)


| Сигнал                | Назначение                                                                       |
|-----------------------|----------------------------------------------------------------------------------|
| `ir_we`               | разрешение записи в `IR`                                                         |
| `pc_inc`              | `PC ← PC + 1`                                                                    |
| `pc_src`              | мультиплексор `PC`: `0=PC+1`, `1=feedback_bus`                                  |
| `a_sel` / `b_sel`     | выбор операндов ALU                                                              |
| `alu_op`              | операция ALU                                                                     |
| `alu_exec`            | строб ALU                                                                        |
| `mar_we`              | `MAR ← ALU_OUT`                                                                  |
| `mem_rd` / `mem_wr`   | строб чтения / записи в память                                                   |
| `mem_byte`            | байтовый режим (`LB`/`SB`)                                                       |
| `check_out`           | разрешение обработки `OUT_PORT`                                                  |
| `reg_we`              | разрешение записи в регистровый файл                                             |
| `reg_src`             | источник `feedback_bus` для writeback                                            |
| `valu_exec`           | строб vector ALU                                                                 |
| `v_reg_we` / `v_reg_src` / `lane_sel` | запись в vector register file (целиком / один lane)        |
| `vbase_we` / `vbase_sel`              | обновление `vbase` (`MAR` или `vbase+1`)                  |
| `addr_sel`            | мультиплексор адреса памяти (`MAR` или `vbase`)                                 |
| `mem_data_src`        | источник данных для `mem_wr` (`rs2` или `VRF`)                                   |
| `br_type` / `addr`    | управление `uPC`                                                                 |
| `halt`                | запрос остановки                                                                 |

### Фазы выполнения инструкций

Архитектура классифицирует инструкции на **скалярные** и **векторные**. Каждая инструкция проходит через набор фаз, определяемых микропрограммой в µROM.

#### Скалярные инструкции

| Фаза | Описание | Инструкции |
| :--- | :--- | :--- |
| **FETCH** | Выборка инструкции из памяти, инкремент `PC`. | Все |
| **ADDR_CALC** | Вычисление эффективного адреса (база + смещение). | `LW`, `SW`, `LB`, `SB`, `B-type` |
| **EXECUTE** | Выполнение операции в АЛУ. | `R-type`, `I-type`, `B-type` |
| **MEM_ACCESS** | Обращение к памяти данных. | `LW`, `SW`, `LB`, `SB` |
| **WRITEBACK** | Запись результата в регистровый файл. | `R-type`, `I-type`, `LW` |

#### Векторные инструкции

Векторные инструкции выполняются как расширенная последовательность микрокоманд, итерируясь по элементам (lanes):

| Фаза | Описание |
| :--- | :--- |
| **V_EXEC** | Выполнение операции в векторной АЛУ (одновременная обработка 4 lanes). |
| **V_LOAD_SEQ** | Последовательный `VLD` (чтение `MEM[vbase+i]` → `VRF[i]`) за 4 такта. |
| **V_STORE_SEQ** | Последовательный `VST` (запись `VRF[i]` → `MEM[vbase+i]`) за 4 такта. |

### Точность моделирования

Машина тактируется по микроинструкциям: каждый вызов `tick` исполняет ровно одну запись из µROM. За такт выбирается микроинструкция, выполняются предписанные ей действия (чтение/запись регистров, памяти, ALU), затем `uPC` переключается на следующую микроинструкцию.

Журнал модели для каждого такта фиксирует:

- номер такта;
- `uPC`, `MIR` (64-битное представление микроинструкции);
- `PC`, `IR`;
- имя микростадии;
- активные управляющие сигналы;
- мнемонику декодированной инструкции;
- значения регистров (`zero`..`t3`, `mar`, `mdr`);
- накопленный выход.
